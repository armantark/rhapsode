Deepest lever

Persist review logs → personal FSRS weights

review_segment already receives a py-fsrs review log on every grade and discards it. New fsrs_review_logs table (card id, FSRS rating, timestamp, duration), rows keyed to the attempt so ⌘Z undo cascades them away — the optimizer never trains on retracted reviews. The fit itself runs as scripts/optimize_fsrs.py via the fsrs[optimizer] extra (torch stays OUT of the app and the PyInstaller sidecar); it writes the fitted parameters to the settings table, and the scheduler reads them from there — defaults until then, per-review thereafter. Same 0.9 target, fewer reviews: this is the "less time" half delivered by FSRS itself.

Fences respected

  • FSRS-pure: nothing schedules outside FSRS; the batch only feeds it better data (personal weights) and exposes what it already knows (due counts, forecast).
  • Practice loop stays light: torch/pandas live only behind the optional optimizer dependency group, invoked by a script — never imported by the app.
  • desired_retention 0.9 unchanged (ruled E1); the retention mirror informs any future change, it doesn't make one.
  • Parked, on purpose: corpus rotation (extra-FSRS reviews), the needed-by date (until the Iliad class deadline hurts), the formula index (until ~5 same-language passages).
