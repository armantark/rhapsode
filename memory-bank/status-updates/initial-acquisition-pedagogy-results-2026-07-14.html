Pattern: provider completed successfully, runner failed to checkpoint/transition.

1. Drain provider stdout and stderr continuously while the child is running; do not wait for process exit before reading a bounded pipe.
2. Atomically persist a valid structured final as soon as it is observed, before any post-process or child-wait step.
3. On resume, salvage a valid final/task_complete from the provider session log or captured stream instead of rerunning a completed write job.
4. Reconcile orphaned running jobs when the recorded child PID no longer exists.
5. Bound or time out post-provider schema validation and record the raw provider result before validation starts.
6. Persist actionable diagnostics: provider exit code, stdout/stderr paths, parser/validation error, last captured event, and whether a valid final was seen.
7. Keep `status` read-only; move dependency repair/install into explicit setup or upgrade commands.

Known policy boundaries

Intentional, bounded behavior

A failed source can add one retry beyond the initial smart-session cap, so a session with many failed fresh lines can grow; each source is still DB-limited to one retry and retries never recurse. In a one-line session, “retry at the tail” is necessarily immediate because no intervening card exists. Both behaviors are visible rather than silently hidden in mastery state.
